# 3.8 Modeling: Model Selection & Hyperparameter Tuning

Welcome to the modeling phase! By now, you've cleaned your data and engineered your features. It is time to pick an algorithm, tune it to perfection, and ensure it can generalize well to data it has never seen before.

This lesson covers the complete step-by-step lifecycle of building a robust machine learning model.

---

## 1. Data Splitting: Train, Validation, and Test Sets

Before doing any modeling, we must split our data. Why? Because testing a model on the same data it learned from is like giving a student the exam questions the night before. They will score 100%, but they haven't actually learned anything!

To prevent this, we split our data into three distinct sets:
1. **Training Set**: The textbook. This is the data the algorithm uses to learn patterns.
2. **Validation Set**: The practice exam. Used to compare different algorithms, tune their settings (hyperparameters), and prevent overfitting. It acts as an unbiased evaluator during the tuning phase.
3. **Test Set**: The final exam. Kept completely hidden until the very end to give us a realistic estimate of the model's true performance.

### How Much Data Goes Where?
The exact percentages depend on how much data you have:
- **Small Datasets (e.g., < 10,000 to 100,000 rows):** Often split **60% Train / 20% Validation / 20% Test**. You need enough data in val/test to accurately judge the model.
- **Large Datasets (e.g., > 1,000,000 rows):** Often split **80% Train / 10% Validation / 10% Test** (or even 98/1/1 for massive deep learning datasets). 10% of a million is still 100k rows, which is plenty for testing!

### Visual Illustration
```text
Total Dataset (100%)
[==================================================]
 \                  |                 |           /
  \                 |                 |          /
[===================|========|========]
   Train (60%)       Val(20%) Test(20%)
```

### Example Code (Getting the 3 splits)

To get three splits in Scikit-Learn, we simply use the `train_test_split` function **twice**.

```python
import numpy as np
from sklearn.model_selection import train_test_split

# Let's assume a small dataset, so we aim for a 60/20/20 split.

# Step 1: First, split off the Test Set (20% of total data)
# We keep this hidden until the absolute end.
X_temp, X_test, y_temp, y_test = train_test_split(X, y, test_size=0.20, random_state=42)

# Step 2: Now we have 80% of our data left in X_temp.
# We want the Validation set to be 20% of the TOTAL data.
# What fraction of 80 is 20? It's 1/4, or 0.25!
# So we split the remaining data, assigning 25% of it to the Validation set.
X_train, X_val, y_train, y_val = train_test_split(X_temp, y_temp, test_size=0.25, random_state=42)

print(f"Training Data (60%): {len(X_train)} samples")
print(f"Validation Data (20%): {len(X_val)} samples")
print(f"Testing Data (20%): {len(X_test)} samples")
```

> **Pros & Cons of a fixed 3-way split:**
> - **Pros:** Simple to implement, fast to execute (only train one model).
> - **Cons:** Sensitive to "lucky" or "unlucky" splits, especially on small datasets. The validation score might not perfectly represent the model's true capability.

## 2. K-Fold Cross-Validation (Model Selection)

Using a single Validation Set is a great start, but it can be risky. What if our random split assigned all the "hard" examples to the validation set by pure chance? Our model would look worse than it is.

**K-Fold Cross-Validation (CV)** solves this. Instead of a single split, we divide the training data into $K$ equal parts (or "folds"). 

### How it works:
If $K=5$:
- We train the model 5 separate times.
- Each time, **1 fold becomes the validation set**, and the **other 4 folds become the training set**.
- We average the 5 different validation scores to get a highly reliable performance metric.

### Math Equation
The final Cross-Validation Score is simply the average of the scores from all $K$ iterations:

$$ CV_{score} = \frac{1}{K} \sum_{i=1}^{K} \text{Score}_i $$

> **Pros & Cons of K-Fold CV:**
> - **Pros:** Provides a much more robust and reliable estimate of model performance because it uses the entire dataset for validation eventually.
> - **Cons:** Computationally expensive! Training takes $K$ times longer because you are training $K$ separate models.

### Auditioning Baseline Algorithms
We use K-Fold CV to "audition" multiple baseline algorithms to see which one naturally fits our problem best before we spend time tuning them.

```python
from sklearn.model_selection import cross_val_score
from sklearn.linear_model import LogisticRegression
from sklearn.ensemble import RandomForestClassifier
from sklearn.svm import SVC

# Define the baseline models we want to test
models = {
    'Logistic Regression': LogisticRegression(),
    'Random Forest': RandomForestClassifier(random_state=42),
    'Support Vector Machine': SVC()
}

# Evaluate each model using 5-Fold Cross Validation
for name, model in models.items():
    # cv=5 means 5-Fold Cross Validation
    scores = cross_val_score(model, X_train, y_train, cv=5, scoring='accuracy')
    
    print(f"{name}:")
    print(f"  Accuracy per fold: {np.round(scores, 3)}")
    print(f"  Average Accuracy: {scores.mean():.3f} (± {scores.std():.3f})\n")
```

## 3. Hyperparameter Tuning with GridSearchCV

Imagine you chose **Random Forest** as your best baseline model. Now, we want to maximize its performance.

In Machine Learning, there are two types of variables:
- **Parameters**: Internal variables the model learns on its own from the data (like the slope $W$ and bias $b$ in Linear Regression).
- **Hyperparameters**: "Knobs" or settings constructed by the human engineer *before* training (like simply how many trees exist in a Random Forest algorithm).

**Grid Search** is a technique that exhaustively tests a "grid" of hyperparameter combinations to find the absolute best setup according to cross validation.

```python
from sklearn.model_selection import GridSearchCV

# Define a grid of hyperparameters to test
param_grid = {
    'n_estimators': [50, 100, 200],   # Number of trees
    'max_depth': [None, 10, 20],      # Maximum depth of the trees
    'min_samples_split': [2, 5]       # Minimum samples required to split a node
}

# The grid has 3 x 3 x 2 = 18 combinations. 
# With 5-Fold CV, this will train 18 x 5 = 90 models!

rf_classifier = RandomForestClassifier(random_state=42)

# Set up GridSearch
grid_search = GridSearchCV(
    estimator=rf_classifier, 
    param_grid=param_grid, 
    cv=5, 
    scoring='accuracy',
    n_jobs=-1 # Uses all CPU cores to speed up the process
)

# Run the search
grid_search.fit(X_train, y_train)

print("Best Hyperparameters found:", grid_search.best_params_)
print("Best CV Score:", grid_search.best_score_)
```

## 4. Retraining on the Full Data

Once we've used our Validation Set (or K-Fold CV) to find the optimal hyperparameters, the best practice is to take that winning configuration and retrain the model on **all** the data we are allowed to touch (Training + Validation). 

Why? Because in Machine Learning, **more data equals a smarter model**. If we have 10,000 rows and hold out 2,000 for validation, our final model should be trained on the full 10,000 before evaluating it on the reserved Test Set.

### Scenario A: Using `GridSearchCV`
If you used `GridSearchCV`, this step is done for you automatically!

> [!TIP]
> **Scikit-Learn makes this automatic!** By default, `GridSearchCV(refit=True)` handles this under the hood. The `grid_search.best_estimator_` is a new model instance that is already re-trained on the entire dataset you passed to `.fit()`.

```python
# The best_estimator_ is ready to use and fully trained on Train + Val data
final_model = grid_search.best_estimator_
```

### Scenario B: What if I didn't use Grid Search?
If you manually tuned your model (e.g., trying a few parameters by hand in a loop) or used a simple predefined Train/Validation split without a grid search object, you must combine the data and retrain it yourself.

**Visual Illustration:**
```text
Step 1: Tune on Train, measure on Val
Train Data (8k) -> Model -> Val Data (2k) (Score: 85%)

Step 2: Combine and Retrain
[Train Data + Val Data (10k)] -> Best Model -> Ready for Test Set!
```

**Example Code:**
```python
# 1. Combine Train and Validation sets back together
# We use np.concatenate for numpy arrays, or pd.concat for pandas DataFrames
X_full_train = np.concatenate((X_train, X_val))
y_full_train = np.concatenate((y_train, y_val))

# 2. Instantiate a fresh model with your manually-discovered best hyperparameters
best_manual_model = RandomForestClassifier(
    n_estimators=100, 
    max_depth=20, 
    min_samples_split=5, 
    random_state=42
)

# 3. Fit it on the combined full training data
best_manual_model.fit(X_full_train, y_full_train)

# Now 'best_manual_model' is our final model, ready for the Test Set!
final_model = best_manual_model
```

## 5. Plotting Learning Curves (Bias vs. Variance)

Before finalizing, we need a "health check" on our model using **Learning Curves**. A learning curve visually tracks how the model's accuracy changes as we feed it more and more training data.

We are checking for two primary diseases:
1. **High Bias (Underfitting)**: The model is too simple. Both the training score and validation score plateau at a very low level. 
2. **High Variance (Overfitting)**: The model memorized the training data! The training score is very high, but the validation score is terrible. There is a massive gap (variance) between the two curve lines.

### Visualizing the Curve

```python
import matplotlib.pyplot as plt
from sklearn.model_selection import learning_curve

# Generate data for the learning curve using our tuned model
train_sizes, train_scores, val_scores = learning_curve(
    final_model, X_train, y_train, cv=5, 
    train_sizes=np.linspace(0.1, 1.0, 5), # Test at 10%, 32%, 55%, 77%, and 100% of data
    scoring='accuracy'
)

# Calculate averages
train_mean = np.mean(train_scores, axis=1)
val_mean = np.mean(val_scores, axis=1)

# Plotting the curves
plt.figure(figsize=(8, 5))
plt.plot(train_sizes, train_mean, 'o-', color='blue', label='Training Accuracy (Memorization)')
plt.plot(train_sizes, val_mean, 'o-', color='orange', label='Validation Accuracy (Generalization)')

plt.title('Learning Curve - Health Check')
plt.xlabel('Number of Training Examples')
plt.ylabel('Accuracy Score')
plt.legend()
plt.grid(True)
plt.show()
```

*Rule of Thumb: If the two lines converge and get close to each other at a high accuracy, your model is perfectly healthy! If they remain far apart, your model is overfitting.*

## 6. Final Evaluation

Finally, we use the **Test Set** that we hid in Step 1. This gives us our most accurate estimate of how our fully-tuned, healthy model will perform in the real world on fresh data.

> [!CAUTION]
> Once you evaluate on the Test Set, **you cannot go back and tweak the hyperparameters!** If you do, the Test Set simply becomes a second Validation Set, and your evaluation is no longer completely trustworthy.

```python
from sklearn.metrics import accuracy_score, classification_report

# Predict on the strictly unseen Test Set
y_pred = final_model.predict(X_test)

# Calculate final Metrics
test_acc = accuracy_score(y_test, y_pred)

print(f"Final Test Accuracy: {test_acc:.2%}")
print("\nDetailed Report:\n", classification_report(y_test, y_pred))
```

## 7. Knowledge Check: Q & A

**Challenge: Take some time to think deeply through these questions before clicking to reveal the answers.**

**Question 1:** Why do we need a completely separate Test Set? Why can't we just use the Validation Set or Cross-Validation fold to report our final accuracy to our boss or stakeholders?
<details>
<summary><b>Click to show answer</b></summary>
Because we made decisions based on the Validation Set (e.g., tweaking hyperparameters multiple times until the validation score looked good). By doing this, we accidentally "leaked" information about the validation data into the model. The Test Set provides the only completely unbiased, untainted measure of how the model performs on data it has intentionally never seen or been optimized against.
</details>

**Question 2:** You ran `GridSearchCV` and subsequently plotted a learning curve. You observe that the Training Accuracy line is nearly 99%, but the Validation Accuracy line is stuck low at 65%. There is a huge gap between them. Are you dealing with High Bias or High Variance? What might be a potential solution?
<details>
<summary><b>Click to show answer</b></summary>
You are dealing with **High Variance (Overfitting)**. The model has essentially memorized the training data but fails completely at generalizing to new inputs. 
To fix it, you could:
1. Provide more training data (which forces the model to generalize).
2. Reduce the complexity of the model (e.g., lower `max_depth` in a tree-based model).
3. Add regularization penalties (for linear algorithms).
</details>

**Question 3:** Assume you are comparing 4 different base algorithms to find the best starting point, and you decide to use 5-Fold Cross Validation. In total, how many separate model configurations will be trained during this phase?
<details>
<summary><b>Click to show answer</b></summary>
20 times. Each of the 4 algorithms must be trained and evaluated 5 times (once for each designated fold). $4 \times 5 = 20$.
</details>

**Question 4:** You employ `GridSearchCV(refit=True)` to discover the best hyperparameters for your Random Forest. Before calling `.predict()` on the Test Set, do you manually need to re-instantiate the `RandomForestClassifier` with those optimal parameters and call `.fit()` again on your training data?
<details>
<summary><b>Click to show answer</b></summary>
No! By setting `refit=True` (which happens to be Scikit-Learn's default), it automatically grabs the smartest hyperparameter configuration and retrains a fresh model on the entire dataset provided during the `grid_search.fit()` step. You can immediately use the resulting object, specifically `.best_estimator_`, to start predicting.
</details>
